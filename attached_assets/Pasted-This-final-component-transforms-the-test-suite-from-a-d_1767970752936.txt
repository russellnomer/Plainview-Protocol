1. Independent Verification (Docker)
We will package the entire testing environment into a Docker container. This ensures that a user in New York runs the exact same test environment as a user in California, eliminating "it works on my machine" discrepancies.

=== FILE: Dockerfile ===
A standalone audit container.

Dockerfile

# Use the official Playwright image (includes browsers)
FROM mcr.microsoft.com/playwright:v1.40.0-jammy

# Set up the working directory
WORKDIR /audit-workspace

# Copy dependency definitions
COPY package.json package-lock.json ./

# Install dependencies
RUN npm ci

# Copy the actual tests and config
COPY . .

# Default command: Run the full suite
CMD ["npx", "playwright", "test"]

# ü§ù Contributing to the Test Suite

We welcome contributions from the community. Whether you found a broken link or want to add a new accountability check, here is how to help.

## The Golden Rule
**Tests must be readable.** We prioritize clarity over clever code. A non-programmer should be able to read the test title and understand what is being verified.

## Adding a New Test
1.  **Duplicate the Template:** Copy `tests/templates/basic_test_template.spec.ts`.
2.  **Name It Clearly:** Use `feature-name.spec.ts`.
3.  **Write the Logic:** focus on user-visible behavior.
4.  **Run It:** `npx playwright test tests/feature-name.spec.ts`

## Naming Conventions
* ‚úÖ `test('Contact form shows error on invalid email')`
* ‚ùå `test('auth_fail_403_regex')`

## Code Review Checklist
* [ ] Does the test fail if I break the feature? (True negative)
* [ ] Is the test name plain English?
* [ ] Did you remove hard-coded wait times (e.g., `waitForTimeout`)?
* [ ] Are screenshots captured on failure?

// 1. Describe the Feature
test.describe('FEATURE NAME (e.g., Donation Page)', () => {

  // 2. Set the Stage
  test.beforeEach(async ({ page }) => {
    await page.goto('/your-page-url');
  });

  // 3. The Test Case

    // A. Locate the element
    const disclaimer = page.locator('.disclaimer-text');
    
    // B. Assert the state

  });

# üõ†Ô∏è Maintenance Guide

## Dependency Schedule
* **Monthly:** Run `scripts/update-deps.sh` to grab the latest browser versions.
* **Quarterly:** Audit `package.json` for unused libraries.

## Version Compatibility Matrix
| Playwright Version | Node Version | Supported Browsers |
|-------------------|--------------|--------------------|
| v1.40+            | 18+          | Chrome 120+, Firefox 121+, Safari 17+ |

## Deprecation Process
If a test is no longer relevant (e.g., a feature was removed):
1.  Do **not** delete the file immediately.
2.  Add a `@deprecated` tag to the test title.
3.  Move the file to `tests/archive/` for historical record.

## Scaling the Suite
As the site grows, run times will increase. To mitigate:
1.  **Shard Tests:** Use Playwright's sharding feature in GitHub Actions to run tests across multiple machines.
    * Command: `npx playwright test --shard=1/4`
2.  **Tagging:** Use tags like `@core`, `@slow`, `@visual` to run specific subsets during development.
